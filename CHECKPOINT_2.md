# Отчет по Контрольной точке №2: Ядро и Инфраструктура

## 1. Текущий статус разработки ядра

В проекте **HealthChecker** реализовано ядро микросервиса для мониторинга доступности сайтов и API.

На текущем этапе реализованы следующие функции:

- загрузка конфигурации из файла `config.yaml`;
- чтение списка URL-адресов для мониторинга;
- периодическая проверка сайтов и API через `HealthCheckWorker`;
- выполнение HTTP-запросов через `HttpClient`;
- проверка HTTP-статуса ответа;
- измерение времени ответа сервиса в миллисекундах;
- проверка SSL-сертификата сайта;
- обработка таймаутов, ошибок подключения и неожиданных HTTP-статусов;
- формирование результата проверки в модели `CheckResult`;
- отправка уведомлений в Telegram при обнаружении ошибки;
- запись событий работы сервиса в логи.

Основные модули проекта:

- `HealthCheckWorker.cs` — запуск проверок по расписанию;
- `EndpointChecker.cs` — проверка URL, HTTP-статуса и времени ответа;
- `SslChecker.cs` — проверка SSL-сертификатов;
- `TelegramNotifier.cs` — отправка уведомлений в Telegram;
- `ConfigLoader.cs` — загрузка конфигурации приложения.

## 2. Интеграция стандартов обслуживания

### Изоляция секретов

В проекте не используются захардкоженные реальные токены и пароли.

Telegram-токен и `chat_id` не хранятся в открытом виде в коде. Для примера безопасной настройки добавлен файл `.env.example`.

Пример содержимого `.env.example`:

```env
TELEGRAM_BOT_TOKEN=your_telegram_bot_token_here
TELEGRAM_CHAT_ID=your_telegram_chat_id_here
```

Файл `.env` добавлен в `.gitignore`, поэтому реальные секреты не попадают в GitHub-репозиторий.

В `.gitignore` указано:

```gitignore
.env
.env.*
!*.env.example
```

Таким образом, в репозитории хранится только пример конфигурации, а реальные значения пользователь задает локально.

---

### Логирование

В проекте настроена система логирования через **NLog**.

Логирование используется в основных сервисах:

- `HealthCheckWorker`;
- `EndpointChecker`;
- `SslChecker`;
- `TelegramNotifier`;
- `ConfigLoader`.

В логах фиксируются:

- запуск приложения;
- загрузка конфигурации;
- начало цикла проверки;
- успешные проверки;
- таймауты;
- ошибки HTTP-запросов;
- проблемы с SSL;
- отправка Telegram-уведомлений.

Используемые уровни логирования:

- `INFO` — успешный запуск, загрузка конфигурации, успешная проверка;
- `WARN` — обнаружена проблема с сайтом или Telegram не настроен;
- `ERROR` — ошибка выполнения запроса или ошибка отправки уведомления.

---

### Контейнеризация

Для контейнеризации проекта добавлен `Dockerfile`.

Проект можно собрать и запустить в Docker-контейнере. Также добавлен файл `docker-compose.yml`, который позволяет запускать сервис через Docker Compose.

Основные команды запуска:

```powershell
dotnet build
dotnet run
docker compose up --build
```
