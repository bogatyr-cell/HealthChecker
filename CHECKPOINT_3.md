## 1. Результаты тестирования и линтинга

### Статический анализ

В проекте **HealthChecker** для статического анализа используется:

- **StyleCop Analyzers** — анализатор стиля и качества C#-кода;
- стандартная проверка сборки через `dotnet build`.

Проверка выполнялась командой:

```powershell
dotnet build
```

Текстовый лог консоли:

```text

    0 Warning(s)
    0 Error(s)

```

Результат: проект успешно собирается, критических ошибок линтера и сборки нет.

### Интерактивная документация

Интерактивная документация Swagger UI / ReDoc для данного проекта не используется, так как **HealthChecker** является фоновым микросервисом на базе `.NET Worker Service`, а не REST API.

Сервис не предоставляет HTTP-контроллеры и публичные REST-эндпоинты. Основная работа сервиса выполняется в фоне: чтение `config.yaml`, проверка URL-адресов, анализ HTTP-статуса, времени ответа, SSL-сертификата и отправка Telegram-уведомлений.

## 2. Инструкция по быстрому запуску

Для запуска проекта с нуля преподавателю нужно выполнить следующие команды.

### 1. Клонировать репозиторий

```powershell
git clone https://github.com/bogatyr-cell/HealthChecker.git
```

### 2. Перейти в папку проекта

```powershell
cd HealthChecker
```

### 3. Создать файл окружения

```powershell
copy .env.example .env
```

Для Linux / macOS:

```bash
cp .env.example .env
```

### 4. Заполнить тестовые значения в `.env`

В файл `.env` нужно вписать тестовые значения:

```env
TELEGRAM_BOT_TOKEN=your_telegram_bot_token_here
TELEGRAM_CHAT_ID=your_telegram_chat_id_here
```

Если Telegram-уведомления не проверяются, можно оставить тестовые значения-заглушки.  
Реальные токены и пароли в репозитории не хранятся.

### 5. Запустить сервис через Docker Compose

```powershell
docker compose up --build
```

После запуска сервис начнет читать настройки из `config.yaml`, проверять указанные URL-адреса и писать результат работы в консоль и лог-файлы.

### Альтернативный локальный запуск без Docker

```powershell
dotnet restore
dotnet build
dotnet run
```
